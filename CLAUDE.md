## What this project is

`spark_az` is a Python library + Synapse notebook for orchestrating child
notebooks with structured Delta-table logging. One orchestrator notebook
calls many children via `mssparkutils.notebook.run`, captures exit values
and exceptions, and writes one queryable row per child to a managed Delta
table. Two delivery formats live side-by-side: an installable wheel
(`src/spark_az/`) and a single self-contained Synapse notebook
(`notebooks/_logging/lgr_inline.ipynb`) that `%run` can pull into any
other notebook with zero install.

The style and conventions mirror the author's companion library
[`github.com/39hops/spark_lib`](https://github.com/39hops/spark_lib).

## What this project is NOT

- **Not a replacement for Synapse pipeline JSON.** The whole point is to
  let one Synapse Notebook activity drive many child notebooks
  sequentially, so the pipeline JSON layer can stay trivial. A future
  spec may add a Python builder for the JSON layer, but it isn't in
  scope here.
- **Not a parallel orchestrator (yet).** `mssparkutils.notebook.run` is
  blocking and v1 is sequential by design. A `run_pipeline_parallel`
  over a thread pool with FAIR-pool support is a v2 follow-on.
- **Not a step-grained logger inside children.** v1 logs one row per
  child notebook, not one per phase inside a child. The
  `pipeline_run_id` propagation contract is designed so a future
  `step()` helper can join into the same table.

## Build / Run

```sh
# Setup — editable install of test + dev extras into the active venv
scripts/setup.sh

# Test — 34 tests, ~10 s after first Spark startup
scripts/test.sh

# Build — produces dist/spark_az-0.1.0-py3-none-any.whl
scripts/build.sh

# Notebook build — regenerate notebooks/*.ipynb from jupytext .py sources
scripts/build_notebooks.sh
```

## Tech stack

- **Language:** Python 3.9+ (Synapse Spark runtimes pin older Python; the
  `typing` module is used everywhere — never PEP 585 builtin generics).
- **Runtime:** PySpark + delta-spark in Synapse. The package imports
  cleanly without pyspark installed (`TYPE_CHECKING`-only imports).
- **Test:** pytest with a local Delta-enabled `SparkSession` fixture
  (`configure_spark_with_delta_pip`) and a `fake_mssparkutils` fixture
  that lets unit tests run without Synapse.
- **Notebooks:** jupytext "percent" `.py` is the source of truth;
  `.ipynb` is generated by `scripts/build_notebooks.sh` and committed.
- **Type checker:** pyright via `pyrightconfig.json` (points at `.venv/`
  for import resolution). `[tool.pyright]` in `pyproject.toml` is the
  fallback for builds without the dedicated file.
- **Key dependencies:** all optional, behind extras. The package itself
  has `dependencies = []`. Extras: `spark` (pyspark + delta-spark),
  `test` (adds pytest), `dev` (jupytext).

## Style

- **Imports:** `from __future__ import annotations` at top of every
  non-trivial `.py`. Imports from the `typing` module only
  (`List`, `Dict`, `Optional`, `Iterable`, `TypedDict`, `TYPE_CHECKING`,
  `Tuple`, `Any`). PySpark imports under `TYPE_CHECKING` at module
  level; real imports inside function bodies.
- **Docs in code:** Google-style docstrings with `Args:`, `Returns:`,
  `Raises:`, `Examples:` sections. **No** inline `#` comments in `.py`
  files. Shebangs in `.sh` and jupytext cell markers
  (`# %%`, `# %% tags=["parameters"]`) are the only exceptions.
- **Naming:** `snake_case` for functions and variables, `PascalCase`
  for classes and TypedDicts, leading underscore for module-private.
- **File size:** keep `src/spark_az/` files under ~500 lines each;
  `lgr.py` is intentionally a single module per the design
  spec (Approach A — refactor into submodules if it grows past ~700).
- **Tests:** `tests/test_<module>.py` for unit, `tests/test_<module>_delta.py`
  for integration tests that need a local SparkSession.

## Handoffs convention

`handoffs/` holds session-to-session notes. Each file starts with a single-line
`Status:` marker:

- `Status: ACTIVE` — work in this handoff is still owed. A new session reads
  the file and executes it (or pushes back if the plan is wrong).
- `Status: COMPLETED <date> — <commit-sha>` — the recommended work shipped in
  that commit. The file is preserved as historical context only.

When you finish work a handoff describes, update its status line to COMPLETED
with the closing commit's SHA. Don't delete old handoffs — the historical
record is the point.

## Specs and plans convention

- `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` — brainstormed design,
  one per feature, locked-decisions table at top, scope boundary at bottom.
- `docs/superpowers/plans/YYYY-MM-DD-<topic>-plan.md` — task-by-task
